## MediaFlow 系统职责边界

> 明确"整理工具 vs 媒体库 vs 播放器"的职责划分，避免功能重复与边界模糊。

---

## 1. 三层架构定位

```
┌─────────────────────────────────────────────────────────────┐
│                     用户/管理员                              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第三层：播放器/客户端                                        │
│  - 浏览器/移动端 App                                         │
│  - 提供播放界面与用户交互                                     │
│  职责：播放、进度记录、收藏、评分                             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第二层：媒体库/媒体服务器                                    │
│  - Jellyfin / Plex / Emby / Kodi                            │
│  - 扫描整理好的目录，提供媒体管理与播放服务                    │
│  职责：扫描、刮削展示信息、转码、播放历史、推荐                │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第一层：整理工具（MediaFlow，我们的系统）                    │
│  - 整理混乱的下载文件，生成规范的目录结构                      │
│  职责：扫描、识别、刮削基础信息、重命名、整理、生成 NFO        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第零层：下载器                                               │
│  - qBittorrent / Transmission / aria2                       │
│  职责：下载文件到原始目录                                     │
└─────────────────────────────────────────────────────────────┘
```

## 2. 职责边界详细划分

### 2.1 整理工具（MediaFlow）的职责

#### ✅ 核心职责（必须做）

1. **文件识别与解析**
   - 扫描下载目录中的文件
   - 从文件名/目录名提取：标题、年份、季集、分辨率、来源、编码等
   - 识别媒体类型：电影/剧集/动漫/纪录片等

2. **刮削基础身份信息**
   - 调用外部 API（TMDB/IMDB/TVDB）获取**最少必需信息**：
     - 标题（title/original_title）
     - 年份（year）
     - 外部 ID（tmdb_id/imdb_id/tvdb_id）
     - 类型/类型（genre，用于二级目录分类）
     - 剧集基本信息（总季数/总集数，用于判断完整性）
   - **不获取**：详细简介、演员、导演、高清海报等（这些交给媒体库）

3. **规则化整理**
   - 按配置的规则生成目标路径（Movies/电影/{Title} ({Year})/）
   - 执行硬链/软链/复制/移动
   - 处理冲突（加后缀/覆盖/跳过）
   - 支持 Dry-run 预览

4. **生成 NFO 文件**
   - 生成符合 Kodi/Jellyfin 规范的 NFO 文件
   - 包含**基础字段**：title/year/外部 ID/genre
   - **不包含**：详细简介、演员列表、高清海报 URL（这些让媒体库自己刮）
   - 目的：让媒体库快速识别，而不是替代媒体库的刮削

5. **防重复与多版本管理**
   - 检测已整理的文件（通过 title+year+season+episode）
   - 识别多版本（同一媒体的不同分辨率/来源）
   - 记录整理历史，支持回滚

6. **审计与任务管理**
   - 记录整理任务（源路径→目标路径）
   - 提供整理状态查询（成功/失败/跳过）
   - 支持重试失败任务

#### ❌ 不做的事情（交给媒体库）

1. **详细展示信息刮削**
   - ❌ 不存储详细简介（plot/overview）
   - ❌ 不存储演员/导演详情
   - ❌ 不下载高清海报/背景图
   - ❌ 不存储用户评分/评论

2. **播放相关功能**
   - ❌ 不提供视频播放
   - ❌ 不记录播放进度
   - ❌ 不管理播放历史

3. **用户个性化功能**
   - ❌ 不提供收藏/心愿单
   - ❌ 不提供推荐系统
   - ❌ 不管理用户观看列表

4. **转码与流媒体**
   - ❌ 不提供视频转码
   - ❌ 不提供流媒体服务
   - ❌ 不管理带宽与质量

### 2.2 媒体库（Jellyfin/Plex/Emby）的职责

#### ✅ 核心职责

1. **扫描与识别**
   - 扫描整理工具生成的规范目录
   - 读取 NFO 文件获取基础信息
   - 或自己重新刮削（如果 NFO 不存在或不完整）

2. **详细刮削**
   - 获取详细简介、演员、导演、制作公司
   - 下载高清海报/背景图/剧照
   - 获取用户评分/评论

3. **媒体管理**
   - 提供媒体浏览界面（按类型/年份/评分筛选）
   - 管理媒体库（添加/删除/刷新）
   - 提供搜索功能

4. **播放服务**
   - 提供视频播放（Web/App/投屏）
   - 转码支持（根据客户端能力）
   - 字幕管理与显示

5. **用户功能**
   - 播放进度记录
   - 收藏/心愿单/评分
   - 播放历史与推荐
   - 多用户管理与权限

#### 🔗 与整理工具的交互

- **输入**：整理工具生成的规范目录 + NFO 文件
- **输出**：无（单向读取，不回传数据给整理工具）

### 2.3 播放器/客户端的职责

#### ✅ 核心职责

1. **播放界面**
   - 提供视频播放界面
   - 控制播放/暂停/快进/快退
   - 显示字幕与音轨选择

2. **用户交互**
   - 浏览媒体库
   - 搜索/筛选
   - 收藏/评分

3. **跨设备同步**
   - 播放进度同步
   - 收藏列表同步

#### 🔗 与媒体库的交互

- **输入**：媒体库提供的 API（获取媒体列表/详情/流媒体地址）
- **输出**：用户操作（播放请求/进度更新/评分等）

## 3. 数据边界

### 3.1 整理工具需要存储的数据

| 数据类型 | 存储位置 | 目的 |
|---------|---------|------|
| **文件路径映射** | 数据库 | 记录源路径→目标路径，支持回滚 |
| **基础身份信息** | 数据库 | title/year/season/episode/外部 ID，用于去重与识别 |
| **技术信息** | 数据库 | 分辨率/来源/编码，用于命名与多版本区分 |
| **类型分类** | 数据库 | genre/category，用于二级目录分类 |
| **整理任务记录** | 数据库 | 任务状态/成功失败数/审计日志 |
| **刮削缓存** | 数据库 | 外部 API 响应缓存，减少重复请求 |
| **基础 NFO** | 文件 | 最小化 NFO（仅 title/year/外部 ID/genre） |

### 3.2 媒体库存储的数据

| 数据类型 | 存储位置 | 目的 |
|---------|---------|------|
| **详细元数据** | 媒体库数据库 | 详细简介/演员/导演/评分 |
| **高清图片** | 媒体库缓存 | 海报/背景图/剧照 |
| **播放记录** | 媒体库数据库 | 播放进度/播放次数 |
| **用户数据** | 媒体库数据库 | 收藏/心愿单/评分 |
| **转码缓存** | 媒体库缓存 | 转码后的视频文件 |

### 3.3 数据不重复存储的原则

**整理工具与媒体库的数据关系**：
- **整理工具**：存储"识别与整理"所需的最小信息
- **媒体库**：存储"展示与播放"所需的完整信息
- **NFO 文件**：作为桥梁，整理工具生成最小 NFO，媒体库读取并补充

**优势**：
1. 避免重复刮削（整理工具刮一次基础信息，媒体库可复用）
2. 职责清晰（整理工具专注整理，媒体库专注展示）
3. 轻量级（整理工具数据库小，启动快）

## 4. 典型工作流

### 4.1 标准流程

```
1. 下载器下载完成
   ↓
2. 整理工具扫描下载目录
   ↓
3. 整理工具识别文件 → 刮削基础信息（title/year/外部 ID/genre）
   ↓
4. 整理工具按规则整理 → 生成规范目录 + 基础 NFO
   ↓
5. 媒体库扫描整理后目录 → 读取 NFO → 补充详细刮削（演员/海报等）
   ↓
6. 媒体库提供播放服务 → 用户通过客户端观看
```

### 4.2 整理工具的输入与输出

**输入**：
- 下载目录中的混乱文件（如 `[Group].Movie.Name.2024.1080p.WEB-DL.x264.mkv`）
- 外部 API（TMDB/IMDB/TVDB，仅获取基础信息）

**输出**：
- 规范的目录结构（`Movies/电影/Movie Name (2024)/Movie.Name.2024.1080p.WEB-DL.x264.mkv`）
- 基础 NFO 文件（仅包含 title/year/外部 ID/genre）
- 整理记录（存入数据库，供审计与回滚）

**不输出**：
- 详细 NFO（让媒体库自己补充）
- 高清图片（让媒体库自己下载）
- 播放服务（完全交给媒体库）

### 4.3 媒体库的输入与输出

**输入**：
- 整理工具生成的规范目录
- 基础 NFO 文件（可选，用于快速识别）

**输出**：
- 完整的媒体库界面（Web/App）
- 视频播放服务（含转码）
- 用户数据（播放进度/收藏等）

## 5. 为什么要这样划分？

### 5.1 单一职责原则

- **整理工具**：只做"整理"这一件事，做到极致
- **媒体库**：专注"展示与播放"，提供完整体验
- **各司其职**：避免功能重复，减少维护成本

### 5.2 解耦与灵活性

- **用户可以选择不同的媒体库**：Jellyfin/Plex/Emby/Kodi 都可以
- **整理工具不绑定特定媒体库**：只生成标准目录 + NFO，兼容所有媒体库
- **媒体库可以独立升级**：不依赖整理工具的数据

### 5.3 性能与资源

- **整理工具轻量级**：只存储必需数据，数据库小，运行快
- **媒体库可以重刮**：如果用户不满意刮削结果，可以让媒体库重新刮削，不影响整理工具
- **避免重复存储**：不在两个地方存储相同的海报/简介等大数据

### 5.4 用户体验

- **整理工具专注整理效率**：快速识别、快速整理、快速生成结果
- **媒体库专注播放体验**：精美界面、流畅播放、个性化推荐
- **各自优化各自的强项**

## 6. 边界案例处理

### 6.1 NFO 文件的"度"

**最小化 NFO 示例**（整理工具生成）：
```xml
<?xml version="1.0" encoding="UTF-8"?>
<movie>
  <title>肖申克的救赎</title>
  <originaltitle>The Shawshank Redemption</originaltitle>
  <year>1994</year>
  <genre>剧情</genre>
  <genre>犯罪</genre>
  <uniqueid type="tmdb">278</uniqueid>
  <uniqueid type="imdb">tt0111161</uniqueid>
</movie>
```

**媒体库会补充**：
- `<plot>`、`<tagline>`、`<rating>`、`<votes>`
- `<actor>`、`<director>`、`<studio>`
- `<thumb>`、`<fanart>`（高清图片 URL）

**为什么不一次性写完整？**
- 整理工具不需要这些信息来完成整理
- 减少整理工具的刮削压力与存储负担
- 让媒体库有机会用自己的刮削源（用户可能更信任某个源）

### 6.2 图片处理

**整理工具**：
- ❌ 不下载海报/背景图
- ❌ 不存储图片 URL（存了也没用，媒体库会重新刮）

**媒体库**：
- ✅ 读取 NFO 中的外部 ID
- ✅ 自己调用 API 获取高清图片
- ✅ 缓存到自己的目录

### 6.3 重复刮削问题

**场景**：整理工具刮了一次 TMDB，媒体库又刮一次，是否浪费？

**解答**：
- **不浪费**：整理工具只刮基础信息（几个字段），请求很轻
- **媒体库可以复用**：如果 NFO 中有外部 ID，媒体库可以直接用 ID 查询，不用再搜索
- **用户可以选择**：如果用户不希望媒体库重刮，可以在整理工具中生成完整 NFO（可选功能）

### 6.4 如果用户不用媒体库？

**场景**：用户只想整理文件，不用 Jellyfin/Plex，直接用本地播放器（如 VLC/PotPlayer）

**方案**：
- 整理工具仍然生成规范目录结构
- NFO 文件可以不生成（可配置）
- 用户直接浏览文件夹，文件名已经很清晰了

## 7. 总结：核心边界原则

| 功能 | 整理工具 | 媒体库 |
|------|---------|--------|
| **扫描文件** | ✅ 扫描下载目录 | ✅ 扫描整理后目录 |
| **识别文件名** | ✅ 解析文件名 | ❌ 依赖 NFO 或重新刮削 |
| **刮削基础信息** | ✅ title/year/外部 ID/genre | ✅ 详细信息（演员/海报等） |
| **重命名整理** | ✅ 核心功能 | ❌ 不做整理 |
| **生成 NFO** | ✅ 最小化 NFO | ✅ 补充完整 NFO（可选） |
| **下载图片** | ❌ 不下载 | ✅ 下载并缓存 |
| **存储详细元数据** | ❌ 不存储 | ✅ 存储 |
| **提供播放** | ❌ 不播放 | ✅ 核心功能 |
| **用户管理** | ❌ 不管理 | ✅ 多用户/权限 |
| **播放记录** | ❌ 不记录 | ✅ 进度/历史 |

**一句话总结**：
- **整理工具**：让文件从"混乱"变"规范"，生成最小 NFO，到此为止
- **媒体库**：读取规范文件，补充完整信息，提供播放服务
